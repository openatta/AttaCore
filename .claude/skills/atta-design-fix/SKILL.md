# 修改方案

> **本阶段目标：** 基于问题报告，定位根因并设计修复方案。只回答"错在哪、怎么改"，不实际改动代码。

## 前置条件

- 问题报告已就绪（由 `/atta-describe-problem` 或 `/atta-bug-fix` 产出）

## 流程

1. **理解问题** — 确认已读完问题报告，理解症状和复现路径
2. **定位根因**（只读代码） — 根据复现路径追踪到相关代码，定位导致问题的具体逻辑
3. **设计修复** — 描述要改什么、为什么这样改
4. **评估影响** — 改动会影响哪些调用方？有没有副作用？是否跨 crate 边界？
5. **给出替代方案** — 至少一个备选，说明优劣

## 铁律

- **只读代码，不改代码** — 这是诊断阶段，不是治疗阶段。可以读文件、追踪调用链，但不能编辑。
- **只动问题相关逻辑** — 不要顺手指出"这里也可以优化"。
- **每条方案有验证标准** — 怎么确认修好了？给出可执行的验证步骤。
- **注意 workspace 边界** — 如果改动跨 `AttaMeta/` 和 `ClawPod/` 两个 workspace，需分别验证构建。

## 产出

输出到对话（不写文件）：

```markdown
## 修复方案：[简短标题]

### 根因分析
- 问题位置: `crates/attacode-*/src/file.rs:行号`
- 原因: ...（为什么当前代码会导致问题报告中的症状）

### 修复路径
- 修改文件: `crates/attacode-*/src/file.rs`
- 修改内容: ...（描述改什么，不给完整代码）
- 理由: ...

### 替代方案
- 方案 B: ...（优劣对比）

### 风险评估
- 副作用: ...
- 影响范围: ...
- 回归风险: 低/中/高

### 验证标准
- 如何确认修复生效: ...
- 原有行为不被破坏: ...
```

## 简报

修复方案输出后，在对话中追加：

```markdown
## 修复方案完成

### 根因
- 位置: `path/file.rs:行号`
- 原因: ...（一句话）

### 修复路径
- 改 `path/file.rs`：...（描述改什么）

### 风险评估
- 副作用: ...
- 回归风险: 低/中/高

### 验证标准
- ...（如何确认修复生效）

### 下一步
→ `/atta-plan-and-execute`（标准：分步实施 + 独立检视）或 `/atta-implement`（快捷：合并最后两步）
```
