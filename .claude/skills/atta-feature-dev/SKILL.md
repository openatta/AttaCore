# 特性开发（简化全流程）

> **本阶段目标：** 端到端完成一个特性的开发——从需求分析到代码交付。适用于不需要独立文档和多人审阅的中等特性。

## 何时用

| 场景 | 用哪个 |
|------|--------|
| 单文件小改动 | 直接干或 `/atta-implement` |
| 中等特性，端到端一步搞定 | `/atta-feature-dev` ← 这个 |
| 跨 crate 大特性，需独立文档和多人审阅 | 完整流程：`/atta-analyze-requirements` → `/atta-design-architecture` → ... |

## 流程

### 阶段 1: 分析与设计（只读）

1. **定义范围** — 做什么、不做什么、前置依赖
2. **梳理场景** — 正常流程 + 异常流程 + 边界条件
3. **设计结构** — crate/模块划分、trait 边界、数据流
4. **关键决策** — 只记结论+理由
5. **分解实施路径** — 粗粒度的 task 列表，让用户能判断工作量

**铁律：不写代码。** 此阶段只读代码（如需），不改文件。

### 阶段 2: 实施决策门

阶段 1 完成后，输出以下简报，**等待用户明确同意**后再进入阶段 3：

```markdown
## [特性名] 特性规格完成

### 范围
- In: ...（2-3 条要点）
- Out: ...

### 结构摘要
- Crate/模块: ... | 数据流: ... | Trait 边界: ...

### 关键决策
- ...（每条一句话）

### 实施路径
- T1: ... — `crates/attacode-*/src/file.rs`
- T2: ... — `crates/attacode-*/src/file.rs`
- ...（预计 N 个 task）

### 风险
- ...

---
**是否开始实施？**（回复"开始"或提出调整意见）
```

### 阶段 3: 实施与收尾（用户确认后）

按照阶段 1 的设计和 task 列表执行：
```
1. 编码 → 优先 TDD（RED→GREEN→REFACTOR）
2. 自检 → 审视改动是否有缺陷
3. 验证 → clippy + test + build
4. 全量验证 + 自审查 + 变更总结
```

## 铁律

- **阶段 1 不改代码** — 只读代码、只出规格。
- **必须等用户确认** — 简报后没有用户明确同意，绝不进入阶段 3。
- **按 task 范围执行** — 不顺手重构、不扩大范围。
- **每次增量可构建** — 一个 task 完成后 clippy + build 必须通过。
- **内嵌检视** — task 自检 + 收尾自审查。发现 Critical 问题 >3 个时停止，退回完整流程。

## 简报（阶段 3 完成后）

```markdown
## [特性名] 开发完成

### 变更清单
- `crates/attacode-*/src/file.rs` — ...

### 验证结果
| 检查项 | 结果 |
|--------|------|
| `cargo nextest run --workspace` | ✅/❌ |
| `cargo clippy --workspace -- -D warnings` | ✅/❌ |
| `cargo fmt --all -- --check` | ✅/❌ |
| `cargo build --workspace` | ✅/❌ |

### 后续
- ...（如有遗留项）
```
