# 需求分析

> **本阶段目标：** 把模糊的用户意图转化为结构化需求规格。只定义"要做什么"，不涉及"怎么做"。

## 流程

1. **理解意图** — 用户的原始输入是什么？期望解决什么问题？
2. **划范围** — 明确 In scope / Out of scope / 依赖条件
3. **梳理场景** — 正常流程、异常流程、边界条件
4. **澄清歧义** — 不确定的地方直接向用户确认，不做猜测
5. **识别风险** — 技术无关的风险：需求自身矛盾、范围蔓延、缺少前置条件

## 铁律

- **禁止读代码** — 此阶段不搜索、不阅读任何源代码文件（项目文档如 CLAUDE.md、docs/ 除外）。需求分析关心的是用户要什么，不是代码里有什么。
- **禁止技术方案** — 不讨论 crate 名、trait 签名、数据结构、文件路径。那是架构设计阶段的事。
- **禁止写代码** — 产出只有需求文档。
- **禁止模糊语** — 不用 TODO / TBD / "完善" / "等"；歧义点当场确认或标记为待澄清。

## 产出

保存为 `docs/reqs/YYYY-MM-DD-[slug].md`：

```markdown
# [功能名] 需求规格

**目标：** 一句话说清要达成什么
**背景：** 为什么需要这个功能

## 范围
- In scope: ...
- Out of scope: ...
- 前置依赖: ...

## 用户场景
### 场景 1: 正常流程
- 给定: ...
- 当: ...
- 则: ...

### 场景 2: 异常流程
- ...

### 场景 3: 边界条件
- ...

## 待澄清
- [ ] ...（如有）

## 风险
- 需求层面风险（非技术风险）
```

## 简报

文档输出后，在对话中输出以下简报：

```markdown
## [功能名] 需求分析完成

**文档：** `docs/reqs/YYYY-MM-DD-[slug].md`

### 范围
- In scope: ...（3-5 条要点）
- Out of scope: ...

### 关键场景
1. 正常: ...（一句话）
2. 异常: ...（一句话）

### 待澄清
- [ ] ...（如有，列出待确认项）

### 风险
- ...（列出需求层面风险，每条一句话）

### 下一步
- 跨 crate / 涉及 trait 边界 / 新增模块 → `/atta-design-architecture`（继续完整流程）
- 中等特性、不需要独立文档 → `/atta-feature-dev`（切换简化全流程）
- 范围明确且很小 → `/atta-implement`（快捷路径）
```
