# 重构优化

> **本阶段目标：** 在保持特性与功能完全不变的前提下，优化软件的非功能质量属性——架构、代码结构、trait 设计、错误处理、死代码清除。不新增功能，不修复业务 bug。

## 何时用

| 场景 | 用哪个 |
|------|--------|
| 优化代码结构、消除技术债、提升可维护性 | `/atta-refactor` ← 这个 |
| 新增功能 | `/atta-feature-dev` 或完整特性 track |
| 修复 bug | `/atta-bug-fix` 或完整修复 track |
| 了解当前代码质量概况 | `/atta-status` |

## 流程

### 阶段 1: 分析与简报（只读，不改代码）

按七个维度审查代码库：

**1. Crate 边界与依赖方向**
- 各 crate 的职责是否清晰、单一？
- 依赖方向是否正确（`attacode-core` 不依赖上层 crate，`attacode-engine` 不依赖 `attacode-tui`）？
- 是否存在跨 workspace 的隐式耦合（`ClawPod/` 和 `AttaMeta/` 之间）？
- Cargo.toml 中 `[dependencies]` 是否最小化？

**2. 模块组织与代码结构**
- 文件组织是否符合 crate 内部约定？
- 模块内聚性：一个 `mod` 是否承担了过多职责？
- 是否存在循环依赖（crate 级别或模块级别）？
- 是否存在过长函数（>80 行）、过大文件（>500 行）、过深嵌套（>4 层）？

**3. Trait 设计与 API 合理性**
- trait 方法签名是否清晰（参数数量、泛型约束、关联类型）？
- `Box<dyn Trait>` 使用是否必要？能否用泛型替代？
- 公共 API 是否遵循最小化原则（`pub` 只暴露必要的）？
- 类型定义是否准确（避免不必要的 `String` 替代 `&str`、正确的 `Option`/`Result` 语义）？

**4. 重复与死代码**
- 是否存在复制粘贴的代码块（>8 行相同逻辑）？
- 是否存在未使用的 `use`、`fn`、`struct`、`const`？
- 是否存在 `#[allow(dead_code)]` 注解掩盖的死代码？
- 是否存在注释掉的代码块？

**5. 错误处理**
- 错误类型定义是否一致（库 crate 用 `thiserror`，应用层用 `anyhow`）？
- `Result<T, E>` 是否正确传播？是否存在 `unwrap()` / `expect()` 可能 panic？
- 错误信息是否有上下文（`.context()` / `.with_context()`）而不是裸字符串？
- 是否存在 `.unwrap()` 或 `let _ =` 吞错误？

**6. 命名与文档**
- 变量、函数、类型命名是否符合 Rust 惯例且意图明确？
- `///` 文档注释是否覆盖公开 API？是否有过时或误导的注释？
- 是否存在魔术数字或魔术字符串？

**7. 性能与 async 模式**
- 是否存在不必要的 `.clone()` / `.to_owned()` / `String` 分配？
- 是否存在同步阻塞操作在 async context（`std::thread::sleep` 在 tokio task 中）？
- 是否存在大 `Arc` / `Mutex` 争用？
- `tokio::spawn` 是否正确处理取消安全？

### 阶段 2: 决策门

阶段 1 完成后，输出以下简报，**等待用户明确同意**后再进入阶段 3：

### 阶段 3: 实施与回归（用户确认后）

对每个重构项执行：

```
1. 理解 → 读相关代码，确认影响范围
2. 重构 → 仅改当前项，保持外部行为完全不变
3. 回归 → clippy + test + build 必须全部通过
4. 等价 → 对比重构前后的测试结果，确认无行为变化
5. 下一个 →
```

**每个重构项完成后项目必须可构建、测试全通过。** 如果某个重构导致测试失败，立即回滚该项并记录原因。

全部完成后输出重构完成简报：

```markdown
## 重构完成简报

### 执行情况
- 计划项: N | 已完成: N | 跳过: N

### 回归结果

|--------|------|
| `cargo nextest run --workspace` | ✅ N passed / ❌ M failed |
| `cargo clippy --workspace -- -D warnings` | ✅ / ❌ |
| `cargo build --workspace` | ✅ / ❌ |

| 重构前 | 重构后 |
|--------|--------|

- [ ] ...（低优先级，可在后续单独处理）
```

## 铁律

- **不新增功能** — 重构只改结构，不改行为。发现缺少的功能记录为建议，不做。
- **不修业务 bug** — 重构中发现 bug 记录并询问用户，不顺手修。避免重构与修 bug 混合导致回归难以定位。
- **每次重构可回归** — 每个重构项完成后 clippy + test + build 必须全通过，否则回滚该项。
- **先分析后动手** — 必须等用户确认简报后才能进入阶段 3。
- **高风险项拆小** — 高风险重构拆成更小步骤（每个步骤单独回归），降低回滚成本。
- **保持等价** — 重构前后的测试结果必须一致。测试通过数减少 = 行为变化 = 不是纯重构。
