# BUG修复（简化全流程）

> **本阶段目标：** 端到端完成一个 bug 的修复——从问题诊断到代码交付。适用于可快速定位的缺陷。

## 何时用

| 场景 | 用哪个 |
|------|--------|
| 一眼能看出问题的小 bug | 直接改或 `/atta-implement` |
| 需诊断的 bug，端到端一步搞定 | `/atta-bug-fix` ← 这个 |
| 复杂 bug、跨多 crate、需独立文档 | 完整流程: `/atta-describe-problem` → `/atta-design-fix` → ... |

## 流程

### 阶段 1: 诊断与方案（只读）

1. **描述症状** — 实际行为 vs 期望行为，复现步骤
2. **定位根因**（只读代码） — 追踪到具体文件和逻辑
3. **设计修复** — 改什么、为什么、副作用评估
4. **给出验证标准** — 怎么确认修好了

**铁律：不改代码。** 此阶段只读代码，可以追踪调用链，不能编辑。

### 阶段 2: 实施决策门

阶段 1 完成后，输出以下简报，**等待用户明确同意**后再进入阶段 3：

```markdown
## [Bug 标题] 修复方案完成

### 症状
- 实际: ... 期望: ...

### 根因
- `crates/attacode-*/src/file.rs:行号` — ...（一句话）

### 修复
- 改 ...：...（描述改什么）
- 副作用: ...
- 回归风险: 低/中/高

### 验证标准
- ...（如何确认修复生效）

---
**是否开始实施修复？**（回复"开始"或提出调整意见）
```

### 阶段 3: 实施与收尾（用户确认后）

```
1. 编码 → 优先 TDD（先写复现测试，确认失败，再修复）
2. 自检 → 审视修复是否正确，有无引入新问题
3. 验证 → clippy + test + build
4. 全量验证 + 变更总结
```

## 铁律

- **阶段 1 不改代码** — 只读代码、出方案。可以追踪调用链，不能编辑。
- **必须等用户确认** — 简报后没有用户明确同意，绝不进入阶段 3。
- **聚焦当前 bug** — 不顺手发现和修复其他问题。
- **先写复现测试** — 修复前必须有失败的测试证明 bug 存在。
- **内嵌检视** — 修复后自审查变更。副作用超过预期时停止，退回完整流程。

## 简报（阶段 3 完成后）

```markdown
## [Bug 标题] 修复完成

### 变更
- `crates/attacode-*/src/file.rs` — ...

### 验证结果
| 检查项 | 结果 |
|--------|------|
| 复现测试 | ✅/❌ |
| `cargo nextest run --workspace` | ✅/❌ |
| `cargo clippy --workspace -- -D warnings` | ✅/❌ |
| `cargo build --workspace` | ✅/❌ |

### 后续
- ...（如有遗留项）
```
