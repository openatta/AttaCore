# 实现对齐审计

> 对照参考实现，审计另一个实现的模块/工具/能力，产出对齐差距报告。适用场景：不同语言重写同一功能、fork 后改造、协议兼容性检查。

## 触发

用户输入 `/atta-compare <feature> [--deep]` 或提出"比较两个实现的 X"、"对齐审计 X"等。

参数：
- `feature`：要比较的功能/模块/工具名（如 `BashTool`、`agent loop`、`permissions`），或 `all`
- `--deep`：深入行为级比较（逐函数/逐分支），不加则为概要级

## 输入

比较前需明确三个要素（从用户参数或上下文推断）：

| 要素 | 说明 | 获取方式 |
|------|------|---------|
| **参考实现 (A)** | ground truth，以它为准 | 用户指定路径，或从项目 CLAUDE.md / memory 推断 |
| **目标实现 (B)** | 待审计实现，差异默认需修 | 用户指定路径，或从工作目录推断 |
| **比较范围** | 哪个模块/工具/概念 | 用户参数 `feature` |

若推断不出，先向用户确认再开始。

## 流程

### Phase 1: 定位代码

根据比较范围，在 A、B 两侧分别定位相关源文件。

定位策略（按优先级）：
1. 按名称搜索：在两侧目录下 grep/glob 匹配的文件
2. 按结构推断：参考侧的目录结构 → 目标侧对应模块
3. 按映射常识：不同语言/框架的惯用文件组织方式

定位完成后，先列出两侧命中的文件清单让用户确认（尤其是 `all` 模式），避免遗漏或误判。

### Phase 2: 读参考实现 (A)

读取参考侧定位到的文件，提炼：

- **公开接口**：函数签名、类型定义、配置项
- **核心逻辑**：主流程分支、状态机、错误处理路径
- **提示词/文本**：给模型的 system prompt、tool description、参数说明、错误信息
- **边界行为**：空值处理、重试策略、超时、取消语义
- **常量/默认值**：关键数值、超时、限制

### Phase 3: 读目标实现 (B)

同 Phase 2，读取目标侧对应文件。

### Phase 4: 逐项比较

每个发现必须标注**两侧的代码位置**（`文件:行`）。

#### 4.1 能力对齐

目标侧是否具备参考侧的全部功能？

```
参考 A: <file>:<line> — <函数/能力>
目标 B: <file>:<line> 或 ❌ 缺失
判定: ✅/⚠️/❌
```

#### 4.2 行为对齐

相同输入是否产生相同输出/副作用？

```
参考 A: <file>:<line> — <逻辑分支/处理方式>
目标 B: <file>:<line> — <对应逻辑>
判定: ✅ 一致 / ⚠️ 语义等价但写法不同 / ❌ 行为偏差
```

#### 4.3 提示词对齐

模型看到的文本是否一致？

```
参考 A: <file>:<line>
  "具体文本..."

目标 B: <file>:<line>
  "具体文本..."

判定: ✅/⚠️/❌ — <差异说明>
```

#### 4.4 流程对齐

核心链路（事件循环、调用链、状态流转）是否一致？

```
参考 A: <file>:<line> — <步骤描述>
目标 B: <file>:<line> — <对应步骤>
判定: ✅/⚠️/❌
```

### Phase 5: 产出报告

```markdown
# 对齐审计报告 — `<feature>` — YYYY-MM-DD

## 比较范围
- 参考实现 (A): `<path>`
- 目标实现 (B): `<path>`
- 涉及文件: A 侧 N 个, B 侧 M 个

## 概要

| 维度 | 评级 | 说明 |
|------|------|------|
| 能力对齐 | ✅/⚠️/❌ | 一句话 |
| 行为对齐 | ✅/⚠️/❌ | 一句话 |
| 提示词对齐 | ✅/⚠️/❌ | 一句话 |
| 流程对齐 | ✅/⚠️/❌ | 一句话 |

## 代码位置清单

### 参考实现 (A)
- `path/to/file.ts` — 做了什么

### 目标实现 (B)
- `path/to/file.rs` — 对应什么

## 详细发现

### ❌ 缺失
| 参考 (A) | 缺失内容 | 影响 |
|-----------|---------|------|
| `file:行` | ... | ... |

### ⚠️ 差异
| 参考 (A) | 目标 (B) | 差异说明 | 判定 |
|-----------|-----------|---------|------|
| `file:行` | `file:行` | ... | 等价/偏差 |

### ✅ 一致
| 参考 (A) | 目标 (B) | 确认点 |
|-----------|-----------|--------|
| `file:行` | `file:行` | ... |

## 提示词差异逐条

（仅当比较涉及 prompt/文本时）

| 参考 (A): 关键文本 | 目标 (B): 对应文本 | 位置 | 判定 |
|---|---|---|---|
| `"..."` | `"..."` | A:`f:l` B:`f:l` | ✅/⚠️/❌ |

## 建议

- **P0 阻塞**: ...
- **P1 重要**: ...
- **P2 改善**: ...
```

## 铁律

- **以参考为准** — 参考实现是 ground truth；差异默认是目标侧需要对齐
- **每条发现必须带位置** — 参考侧 `文件:行` + 目标侧 `文件:行`（缺失则标 ❌）
- **标注语义等价** — 实现方式不同但语义等价时标 ⚠️ 并说明为什么等价
- **不修改代码** — 本 skill 只审计、报告
- **提示词逐字比** — prompt 文本逐词差异都可能影响模型行为
- **不确定就标 ❓** — 不强行给结论

```markdown
| 流程 | ✅/⚠️/❌ |

- ⚠️ M 项差异

- B: `path/x.rs` `path/y.rs`
```
