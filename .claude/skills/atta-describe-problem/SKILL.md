# 问题说明

> **本阶段目标：** 把"出了个 bug"转化为精确、可复现的问题报告。只描述现象，不做诊断。

## 流程

1. **收集症状** — 用户看到了什么？期望看到什么？实际发生了什么？
2. **确定复现路径** — 精确的操作序列，标注关键参数
3. **记录环境** — 操作系统、版本、相关配置
4. **评估影响** — 是否阻塞？有无 workaround？影响哪些用户/流程？
5. **缩小范围** — 是否最近才出现？与某次变更相关？（只问不查代码）

## 铁律

- **禁止查代码** — 此阶段不搜索、不阅读源代码。问题说明是症状描述，不是诊断。
- **禁止猜测根因** — 不写"可能是 X crate 的问题"。那是修改方案阶段的事。
- **禁止提修复方案** — 不写"改一下 Y 就好了"。那是修改方案阶段的事。
- **必须可复现** — 没有复现步骤的问题报告无法验证修复。

## 产出

输出到对话（不写文件）：

```markdown
## 问题报告：[简短标题]

### 症状
- 实际行为: ...
- 期望行为: ...
- 出现频率: 每次 / 间歇 / 仅一次

### 复现步骤
1. ...
2. ...
3. 观察到错误: ...

### 环境
- OS: ...
- 版本/分支: ...
- 相关配置: ...

### 影响评估
- 严重程度: 阻塞 / 严重 / 轻微 / 纯外观
- 影响范围: ...
- 临时绕过: 有/无（描述）

### 关联线索
- 最近变更: ...（如有线索）
- 首次发现: ...
```

## 简报

问题报告输出后，在对话中追加：

```markdown
## 问题报告完成

### 症状摘要
- [标题]：实际 ... 期望 ...

### 复现路径
1. ...（最简复现步骤）

### 影响
- 严重程度: 阻塞 / 严重 / 轻微 / 纯外观
- 临时绕过: 有/无

### 下一步
→ `/atta-design-fix`（标准：继续完整流程）或 `/atta-bug-fix`（切换简化全流程）
```
